# NestJS Monorepo (Microservices + gRPC)

This repository information about how to setup a **NestJS monorepo setup** structured into several microservices that share a **common contract library** and **gRPC `.proto` definitions**.

See the **SETUP.md** file for instructions.

---

## 📁 Organizing .proto Files

All gRPC .proto files are placed in the shared proto/ folder at the project root:
```bash
./proto/
  ├── common.proto
  ├── users.proto
  ├── orders.proto
  └── notifications.proto
```

1. Configure `nest-cli.json` to include `proto`

Ensure proto files are copied to `dist` by adding them to `assets`:
```bash
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "src",
  "monorepo": true,
  "compilerOptions": {
    "tsConfigPath": "tsconfig.json",
    "assets": [
      "proto"       // copies proto/ to dist/
    ],
    "watchAssets": true
  },
  "projects": {
    "api-gateway": {
      "type": "application",
      "root": "apps/api-gateway",
      "entryFile": "main"
    },
    "users": {
      "type": "application",
      "root": "apps/users",
      "entryFile": "main"
    },
    "orders": {
      "type": "application",
      "root": "apps/orders",
      "entryFile": "main"
    }
  }
}
```

Now files like `proto/users.proto` are copied to `dist/proto/users.proto` automatically on build.

2. Use `.proto` in a microservice
Example configuration in a gRPC microservice (e.g. `users`) located in `apps/users/src/main.ts`:

```bash
import { NestFactory } from '@nestjs/core';
import { MicroserviceOptions, Transport } from '@nestjs/microservices';

async function bootstrap() {
  const app = await NestFactory.createMicroservice<MicroserviceOptions>(
    AppModule,
    {
      transport: Transport.GRPC,
      options: {
        package: 'users',
        protoPath: './proto/users.proto',  // points to dist/proto/users.proto
        url: '127.0.0.1:50051',
      },
    },
  );
  await app.listen();
}
bootstrap();
```

## 🧩 Shared Contracts Library
DTOs, interfaces, and types shared across microservices and the gateway go into the `contracts` library:
```bash
nest generate library contracts
```

Then remove the boilerplate code generated by the CLI:
```bash
rm -rf libs/contracts/src/*
```

Example folder structure
```bash
libs/
└── contracts/
    └── src/
        └── users/
            ├── user.dto.ts
            ├── user.request.dto.ts
            └── user.response.dto.ts
        └── orders/
            ├── order.dto.ts
            └── order.response.dto.ts
```

- Each microservice and the gateway can import DTOs from `@project_name/contracts/...`.
- These DTOs can be aligned with, or generated from, the `.proto` files, but gateway‑specific DTOs should not be shared.

## 🏁 Vision
This repository implements a clean, scalable monorepo structure where:
- The API gateway exposes HTTP APIs.
- Microservices handle domain logic via gRPC.
- `.proto` files in `proto/` define contracts.
- Shared DTOs live in `libs/contracts/`.

These patterns make it easier to maintain type‑safe, loosely‑coupled services for a project.
