# NestJS Microservice Setup (Monorepo Structure)

This guide describes the step‑by‑step creation of a **NestJS monorepo project** with multiple microservices sharing a common **contracts library**.

---

## 🚀 1. Create a Standard NestJS Project

- Start by creating a basic NestJS application using the CLI:
```bash
nest new projet_name
```

- Move into the project directory:
```bash
cd projet_name
```

## 📁 2. Convert the Project into a Monorepo
Next, convert this standard project into a monorepo structure to host multiple microservices.

From the root of the project, generate your first app (for example, an API gateway):
```bash
nest generate app api-gateway
```

You can then generate additional microservices as needed, for example:
```bash
nest generate app users
nest generate app orders
nest generate app notifications
```
Each generated app will be located under the `apps/` folder (`apps/api-gateway`, `apps/users`, etc.).

## 📁 3. Configure the `nest-cli.json` File
Open the `nest-cli.json` file at the project root and remove all references to the original default application that was created with `nest new projet_name`.

Ensure that the configuration points explicitly to your new apps. For example:
```bash
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "apps",
  "monorepo": true,
  "compilerOptions": {
    "tsConfigPath": "tsconfig.json"
  },
  "projects": {
    "api-gateway": {
      "type": "application",
      "root": "apps/api-gateway",
      "entryFile": "main",
      "sourceRoot": "apps/api-gateway/src"
    },
    "users": {
      "type": "application",
      "root": "apps/users",
      "entryFile": "main",
      "sourceRoot": "apps/users/src"
    }
  }
}
```

By default, NestJS will start the API gateway (the first app in the config). You can change this behavior by using the `--app` flag when starting the app, for example:
```bash
nest start --app api-gateway
```

## 🧩 4. Create a Shared Contracts Microservice Library
To standardize communication between microservices, create a shared contracts library that defines the messages, their payloads, and expected responses.

Generate a new library:
```bash
nest generate library contracts
```
This will create a library under `libs/contracts`.

Next, remove the boilerplate code generated by the CLI:
```bash
rm -rf libs/contracts/src/*
```

Now, create a domain‑specific folder inside the library, for example for the `users` microservice:
```bash
mkdir -p libs/contracts/src/users
```

Inside libs/contracts/src/users, add your DTOs and interfaces, for example:
- `user.dto.ts`
- `user.request.dto.ts`
- `user.response.dto.ts`

Example structure:
```bash
libs/
└── contracts/
    └── src/
        └── users/
            ├── user.dto.ts
            ├── user.request.dto.ts
            └── user.response.dto.ts
```

## 🏁 5. Share Contracts Without Exposing Gateway DTOs
A contract is a set of rules defining how to communicate with a given microservice application or a generic application. Each microservice will define what messages it can handle, what payloads it expects, and what data it returns. These contracts are placed in the shared library and used by both the microservices and the API gateway.

This shared approach ensures:
- Typesafety between services.
- Reduced duplication of DTOs.
- Easier refactoring (centralized contract definitions).

You do not need to share DTOs specifically tied to the gateway itself (UI‑only DTOs, views, responses meant for HTTP only).

## ⚙️ 6. Useful Commands
Generate a module for a given app in the monorepo, for example for the gateway:
```bash
nest generate module users --project api-gateway
```

You can also generate services, controllers, and other components within specific apps:
```bash
nest generate service users --project users
nest generate controller users --project users
```

List all projects in the monorepo:
```bash
nest list
```

Start a specific app:
```bash
nest start --app api-gateway
```

You now have a solid foundation to build a NestJS monorepo project with microservices and a shared contracts library for clean, scalable, and type‑safe communication.
